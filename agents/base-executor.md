# Base Executor Template

本文件定义 Ralph Loop Task Executor 的通用执行指令。所有 agent 特定的 executor 都应继承此模板。

## 角色定义

**Task Executor** 负责增量执行 features.json 中定义的功能。

## 核心原则（所有 Agent 必须遵守）

### 六大铁律

| # | 原则 | 说明 |
|---|------|------|
| 1 | **增量工作** | 每次只完成一个 `passes: false` 的功能 |
| 2 | **干净状态** | 完成后确保代码可提交（无语法错误、测试通过） |
| 3 | **只修改 passes** | 只能将 `passes: false` 改为 `passes: true`，禁止修改其他字段 |
| 4 | **端到端验证** | 使用浏览器自动化工具验证功能 |
| 5 | **提交进度** | 完成后 git commit 并更新 progress.md |
| 6 | **完成信号** | 必须单独一行输出 `MISSION_COMPLETE` |

### 禁止事项

- ❌ 不要一次实现多个功能
- ❌ 不要删除或修改 features.json 中的测试步骤
- ❌ 不要在未测试的情况下标记 `passes: true`
- ❌ 不要留下未提交的代码
- ❌ 不要修改已跳过的功能（达到最大重试次数）
- ❌ 不要跳过需求文档阅读

## 通用执行流程

```
┌─────────────────────────────────────────────────────────────┐
│  1. 获取上下文                                                │
│     - 当前目录                                                │
│     - Git 历史 (最近 10 条)                                   │
│     - features.json                                          │
│     - task.md（包含需求文档引用）                              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  2. 选择第一个 passes: false 的功能                           │
│     - 如果所有功能都 passes: true，任务完成                    │
│     - 如果功能被跳过（重试次数达到上限），跳过                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  3. 阅读需求文档（重要！）                                     │
│     - 检查 requirement_refs 字段                              │
│     - 如有，读取引用的需求文档                                 │
│     - 理解完整业务规则后再开始实现                              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  4. 验证基础功能                                              │
│     - 使用浏览器自动化测试基本功能                             │
│     - 发现现有 bug 先修复                                     │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  5. 实现单个功能                                              │
│     - 编写代码                                                │
│     - 运行单元测试                                            │
│     - 端到端验证（verify_command 或浏览器自动化）              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  6. 更新状态                                                  │
│     - 只修改 passes: false → passes: true                    │
│     - 不要删除或修改其他内容                                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  7. 提交进度                                                  │
│     - git add -A                                             │
│     - git commit -m "feat: 完成功能 XXX"                      │
│     - 更新 progress.md                                       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  8. 输出完成信号                                              │
│     MISSION_COMPLETE                                         │
└─────────────────────────────────────────────────────────────┘
```

## 验证流程

### 三层验证（stop-hook.sh）

Ralph Loop 在每次迭代结束时执行三层验证：

1. **MISSION_COMPLETE 信号检查**
   - 确认 AI 输出了完成信号
   - 未输出则视为任务失败

2. **Git 状态检查**
   - 确保工作区干净（无未提交更改）
   - 有更改则视为任务失败

3. **自定义验证命令**
   - 执行功能的 `verify_command`
   - 命令失败则重试该功能

### 重试机制

- 每个功能最多重试 `MAX_FEATURE_RETRIES` 次（默认 3 次）
- 达到重试上限后自动跳过该功能
- 跳过的功能会在 features.json 中标记

## features.json 操作规范

### 只允许的修改

```json
// 修改前
{
  "id": "F001",
  "description": "功能描述",
  "verify_command": "npm test",
  "passes": false    // ← 只能修改这个
}

// 修改后
{
  "id": "F001",
  "description": "功能描述",
  "verify_command": "npm test",
  "passes": true     // ← 修改为 true
}
```

### 禁止的修改

- ❌ 删除功能
- ❌ 修改 `id`
- ❌ 修改 `description`
- ❌ 修改 `steps`
- ❌ 修改 `verify_command`
- ❌ 修改 `requirement_refs`

## 端到端测试

### 浏览器自动化工具

所有主流 AI Agent 都支持浏览器自动化：

| Agent | 浏览器工具 |
|-------|-----------|
| Claude Code | `mcp__chrome-devtools__*` |
| Codex CLI | 内置浏览器集成 |
| Gemini CLI | `browser_agent` |
| OpenClaw | 待确认 |

### 测试场景

1. **导航测试** - 页面加载、路由跳转
2. **交互测试** - 点击、填写表单、提交
3. **视觉验证** - 截图对比、快照检查
4. **数据验证** - 数据展示、状态更新

## 错误处理

### 遇到错误时

1. **记录错误** - 在 progress.md 中记录错误详情
2. **分析原因** - 理解错误根因
3. **尝试修复** - 修复后重新验证
4. **请求帮助** - 无法解决时向用户求助

### 重试策略

- 自动重试：网络错误、临时故障
- 手动干预：逻辑错误、需求不明确

## 进度记录

### progress.md 格式

```markdown
## 2024-01-15 14:30

### 完成功能
- F001: 用户登录 UI

### 验证结果
- 单元测试: 通过
- E2E 测试: 通过

### 提交
- commit: abc123 "feat: 完成用户登录 UI"

### 备注
- 修复了表单验证的边界情况
```

## 扩展执行指令

每个 agent 特定的 executor 应包含：

1. **代理信息** - 名称、配置文件、特点
2. **核心工具** - 可用的工具列表
3. **语法特点** - 工具调用的语法格式
4. **工作流适配** - 将通用流程映射到特定工具
5. **工具调用示例** - 具体的语法示例
6. **配置文件示例** - 项目根目录的配置文件格式
